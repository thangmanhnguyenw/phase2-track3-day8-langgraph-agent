🎯 Dự án này làm gì? (giải thích 30 giây)

Hãy tưởng tượng bạn là tổng đài viên AI nhận ticket hỗ trợ khách hàng. Mỗi tin nhắn đi qua dây chuyền xử lý (graph): tiếp nhận → phân loại (LLM) → xử lý theo loại → trả lời → ghi log. Một số loại cần hỏi thêm, gọi tool, xin duyệt người, hoặc thử lại khi lỗi.
🧠

LangGraph StateGraph

Workflow dạng đồ thị: mỗi bước là 1 node, state được truyền và cập nhật qua từng node.

🔀

Conditional Routing

Sau classify, graph rẽ nhánh theo route — không phải query nào cũng đi cùng đường.

🔄

Retry Loop (có giới hạn)

Lỗi tạm thời → thử lại tối đa max_attempts lần → hết thì dead letter.

👤

HITL (Human-in-the-Loop)

Hành động nguy hiểm (refund, xóa account) phải qua node approval trước khi thực thi.

🔁 Luồng chạy tổng — tất cả bắt đầu giống nhau

Giai đoạn 1: Mọi query đều đi qua

START → intake
chuẩn hóa text → classify
LLM phân loại → route_after_classify
rẽ 5 nhánh

Sau classify, state có route = một trong: simple | tool | missing_info | risky | error

Giai đoạn 2: 5 nhánh xử lý

simple answer→ LLM trả lời→ finalize → END
tool tool→ evaluate→ retry?→ answer→ finalize
missing_info clarify→ hỏi thêm user→ finalize
risky risky_action→ approval→ tool→ evaluate→ answer→ finalize
error retry→ attempt < max?→ tool→ evaluate→ dead_letter hoặc answer→ finalize

Retry loop giải thích đơn giản

                    ┌──────────────────────────────────┐
                    │         evaluate_node            │
                    │  tool result có "ERROR"?         │
                    └────────────┬─────────────────────┘
                          Có     │     Không
                           ▼     │      ▼
                        retry    │    answer
                    (attempt++)  │
                           ▼     │
              attempt < max_attempts?
                    │         │
                   Có        Không
                    ▼         ▼
                  tool    dead_letter
                    │
                    └──► evaluate (lặp lại)

📋 Luồng chi tiết — 7 Scenarios (từng bước một)

Mỗi scenario = 1 ticket khách hàng. Dưới đây là chính xác các node graph sẽ đi qua.

S01_simple
"How do I reset my password?"
route: simple
  1. intake — Chuẩn hóa câu hỏi
  2. classify (LLM) — Nhận diện: câu hỏi FAQ, không cần tool → route=simple
  3. answer (LLM) — Sinh hướng dẫn reset password
  4. finalize — Ghi log hoàn thành → END
intake→ classify→ answer→ finalize

✅ Đường ngắn nhất — 4 nodes. Không retry, không approval.

S02_tool
"Please lookup order status for order 12345"
route: tool
  1. classify (LLM) — Tra cứu thông tin → route=tool
  2. tool — Mock API trả về: "Order 12345 status SHIPPED"
  3. evaluate — Không có ERROR → evaluation_result=success
  4. answer (LLM) — Trả lời khách dựa trên kết quả tool
  5. finalize → END
intake→ classify→ tool→ evaluate→ answer→ finalize
S03_missing
"Can you fix it?"
route: missing_info
  1. classify (LLM) — Câu quá mơ hồ, thiếu context → route=missing_info
  2. clarify — Hỏi lại: "Bạn cần fix cái gì? Account/order nào?"
  3. finalize → END (không đoán bừa / hallucinate)

💡 Tại sao quan trọng: Agent production không được bịa câu trả lời khi thiếu thông tin.

S04_risky
"Refund this customer and send confirmation email"
route: risky + HITL
  1. classify (LLM) — Refund + email = side-effect → route=risky
  2. risky_action — Ghi proposed_action cần duyệt
  3. approval — Mock approve (approved=true). Metrics ghi interrupt_count=1
  4. tool — Thực thi sau khi được duyệt
  5. evaluate → answer (LLM) → finalize

🔒 requires_approval=true → metrics kiểm tra approval_observed=true

S05_error
"Timeout failure while processing request"
route: error + retry
  1. classify (LLM) — Lỗi hệ thống → route=error
  2. retry — attempt=1, ghi lỗi transient
  3. tool — attempt<2 → trả về chuỗi chứa "ERROR" (mô phỏng lỗi)
  4. evaluate — Phát hiện ERROR → needs_retry
  5. retry — attempt=2
  6. tool — attempt≥2 → SUCCESS lần này
  7. evaluate → answer (LLM) → finalize

🔄 Retry loop chạy 1–2 vòng rồi thành công. total_retries trong metrics tăng.

S06_delete
"Delete customer account after support verification"
route: risky + HITL
  1. classify (LLM) — Xóa account = destructive → route=risky
  2. risky_action → approval → tool → evaluate → answer → finalize

Giống S04 nhưng hành động nguy hiểm hơn (xóa dữ liệu vĩnh viễn).

S07_dead_letter
"System failure cannot recover after multiple attempts"
max_attempts=1
  1. classify (LLM) → route=error
  2. retry — attempt tăng lên 1
  3. route_after_retry — 1 < 1? Không → đi dead_letter ngay
  4. dead_letter — Trả final_answer: "đã escalate, không xử lý được"
  5. finalize → END

☠️ Kiểm tra graph không loop vô hạn — điểm quan trọng trong rubric.

⚙️ Chức năng từng Node — đọc 1 lần là hiểu

NodeInput đọc từ stateOutput ghi vào state1 câu giải thích
intake query query (trim), messages, events Làm sạch text đầu vào trước khi xử lý.
classify LLM query route, risk_level, events AI đọc câu hỏi và quyết định "loại ticket" — không hard-code keyword.
tool route, attempt, query tool_results, events Giả lập gọi API/API ngoài; route=error + attempt<2 thì trả ERROR.
evaluate tool_results[-1] evaluation_result, events Kiểm tra tool có thành công không — cổng quyết định retry.
answer LLM query, tool_results, approval final_answer, events AI viết câu trả lời cuối, chỉ dựa trên context đã có.
clarify query pending_question, final_answer, events Hỏi user thêm thay vì đoán.
risky_action query proposed_action, events Mô tả hành động nguy hiểm cần duyệt.
approval HITL proposed_action approval{approved, reviewer}, events Người duyệt (hoặc mock) cho phép / từ chối hành động.
retry attempt attempt+1, errors, events Đếm số lần thử, ghi log lỗi tạm thời.
dead_letter attempt, errors final_answer, events Bỏ cuộc, escalate — hàng đợi "không cứu được".
finalize toàn bộ state events Điểm kết thúc bắt buộc — mọi nhánh phải đến đây.

📦 State — "Bộ nhớ" graph mang theo suốt hành trình

LangGraph không dùng biến global. Mỗi lần chạy có 1 object AgentState — như chiếc cặp node nào mở ra, bỏ thêm/thay đồ vào, truyền sang node tiếp.

Append (cộng dồn — không xóa cũ)

messages, tool_results, errors, events — dùng Annotated[list, add]

Overwrite (ghi đè — chỉ giữ giá trị mới nhất)

route, attempt, evaluation_result, approval, final_answer

}
